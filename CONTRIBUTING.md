# Contributing to MotorsportVerse

Thanks for helping build the open-source motorsport AI ecosystem.

## Ways to contribute

1. **Add a sport** — the highest-leverage contribution. See
   [docs/adding-a-sport.md](docs/adding-a-sport.md).
2. **Improve the shared packages** — `motorsport-core` / `motorsport-data`.
3. **Improve the ecosystem website** — catalog UX, project pages.
4. **Register a concept** — add a `registry/projects/<slug>.json` entry for a
   sport you'd like to see.

## Development setup

```bash
pip install -e "packages/motorsport-core[dev]" "packages/motorsport-data[dev]"
cd website && npm install
```

## Before opening a PR

```bash
# Python
ruff check packages projects scripts
pytest packages/motorsport-core packages/motorsport-data projects/f2-predictions

# Registry (if you touched registry/projects/*)
python scripts/build_registry.py

# Website (if you touched website/)
cd website && npm run build
```

CI runs exactly these. PRs must be green.

## Conventions

- **Python**: 3.11+, `ruff` (E/F/W), line length 100. Keep `motorsport-core`
  dependency-light (no sport constants, no web/plotting in the import path).
- **Leakage discipline**: any multi-round aggregation must filter to rounds
  strictly before the target round and assert it with
  `motorsport_core.leakage.assert_prior_only`.
- **Schema changes**: update `motorsport_data.schema` and the TS mirror in
  `website/src/types/registry.ts` together; prefer additive changes.
- **The registry is the source of truth** for the catalog. Don't hand-edit
  `registry/index.json` or `website/public/data/registry.json` — they're
  generated by `scripts/build_registry.py`.
- **F1 is the reference.** When a design question has no obvious answer, match
  what the flagship `f1_predictions` project does.

## Adding a shared-core module

If you extract more reusable logic from a project into `motorsport-core`, add it
to `__init__.__all__`, give it a test under `packages/motorsport-core/tests/`,
and document it in [docs/core-api.md](docs/core-api.md).
